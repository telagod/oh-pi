# Ant Colony for Pi：产品边界

> 这份文档回答四个问题：它是什么、适合谁、什么时候用、当前阶段不做什么。

## 1. 产品定义

**Ant Colony for Pi** 是一个为 `pi-coding-agent` 设计的多智能体协作插件。

它的目标不是替代 pi，而是补强 pi 在复杂任务下的执行层能力，让单代理难以稳定处理的工作，能够被拆解为：

- 侦察（scouting）
- 规划（planning）
- 并行执行（working）
- 审查与回退（reviewing / repair）
- 汇总（reporting）

当前仓库仍保留 `oh-pi` 作为分发与引导入口，但长期方向是：

- **colony 是主产品**
- **oh-pi 是 bootstrap 层**

---

## 2. 目标用户

优先服务这些用户：

- 已经在使用 pi
- 遇到多文件、跨模块、可并行的复杂编码任务
- 希望降低人工拆任务与反复审查的负担
- 需要一个比单 agent 更稳的执行流

不优先服务：

- 只想快速装主题/预设的用户
- 主要做问答式使用的用户
- 很少遇到复杂工程任务的轻度用户

---

## 3. 适用场景

以下场景优先使用 colony：

### 3.1 多文件修改
例如：
- 从 session auth 重构为 JWT
- 新增 feature 并补齐 service / route / test
- 统一迁移 API 调用模式

### 3.2 跨模块重构
例如：
- 重命名核心抽象并修复依赖面
- 拆分 monolithic module
- 增加 adapter / store / runtime 边界

### 3.3 可并行工程任务
例如：
- 一部分 worker 改实现
- 一部分 worker 补测试
- reviewer 统一做质量守门

### 3.4 需要审查闭环的任务
例如：
- 先执行，再统一校验差异
- 需要对失败任务进行恢复或重试
- 需要知道哪些任务适合继续并行、哪些应该串行

---

## 4. 不适用场景

以下场景默认不该启用 colony：

- 只改 1 个文件的小修复
- 快速问答、解释代码、生成一次性脚本
- 需要你人工逐步确认每一步的任务
- 上下文高度集中、几乎无法拆分的问题
- 并行收益低于协调成本的问题

一句话：

> **如果协调成本大于拆解收益，就不该启用蚁群。**

---

## 5. 产品价值，不是“仿生概念”，而是执行收益

我们真正要证明的，不是“蚁群这个概念很酷”，而是它在特定任务上能带来这些收益：

- 更好的任务拆解
- 更高的多文件吞吐
- 更低的遗漏率
- 更强的执行后审查能力
- 更清楚的失败恢复路径

仿生只是解释层，不是主价值层。

---

## 6. 当前阶段的产品原则

### 原则 1：插件优先
尽量以 pi extension 的方式存在，而不是深度改造宿主。

### 原则 2：边界优先
在继续扩功能前，优先稳定这些边界：
- `PiAdapter`
- `PheromoneStore`
- colony runtime / pi integration 分层

### 原则 3：证据优先
每次强调 colony 能力时，都要回答：
- 比单 agent 好在哪里？
- 好多少？
- 在什么任务上好？
- 在什么任务上反而不该用？

### 原则 4：少而强
宁愿把少数任务场景做深，也不追求“所有任务都能开蚁群”。

---

## 7. 当前阶段不做什么

- 不把产品继续扩展为大而全配置包
- 不优先做分布式集群
- 不优先做复杂商业化功能
- 不优先扩展过宽的社区叙事
- 不把主题、预设、技能包装当作主卖点

---

## 8. 未来结构方向

更理想的结构会逐渐演进为：

```txt
packages/
  ant-colony-core/
  ant-colony-pi/
  ant-colony-bench/
  oh-pi-bootstrap/
```

即使暂时不拆仓库，也应先在逻辑上做到：

- colony core 独立
- pi integration 独立
- bootstrap / installer 独立

这样产品和工程边界才会一致。
